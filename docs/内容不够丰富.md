# 内容不够丰富

## 现象

生成的 PPT 要点偏短、偏泛，缺少数据、案例和层次；多页版式相似，读起来像提纲而不是可讲的演示稿。

## 根因

| 环节 | 问题 |
|------|------|
| 生成流程 | 原先仅 **一次** LLM 调用，模型同时兼顾结构、版式与正文，容易牺牲深度 |
| Prompt | 要求「3–5 条 **concise** bullet」，显式鼓励简短概括 |
| 与目标流程不一致 | [01.PPT生成流程.md](./01.PPT生成流程.md) 规划为「大纲 → 初稿 → 润色」，代码未落地 |

## 方案（已实现）

`AiService.generateContent` 改为三阶段流水线：

```mermaid
flowchart LR
  A[用户 topic + slideCount] --> B[generateOutline]
  B --> C[generateSlidesFromOutline]
  C --> D[polishContent]
  D --> E[PresentationContent]
```

1. **大纲**（`generateOutline`）：每页 purpose + 4–6 条带具体信息的 keyPoints，并建议 layout。
2. **扩写**（`generateSlidesFromOutline`）：按大纲输出完整 JSON；每页 **4–6 条、每条 20–50 词**，禁止空泛套话。
3. **润色**（`polishContent`）：在保持页数与 layout 不变的前提下补数据、去重、平衡双栏与图表。

相关代码：`api/src/ai/ai.service.ts`、`api/src/ai/types/presentation-outline.ts`。

## 代价与注意

- **延迟与费用**：约 3 倍 LLM 调用（每次最长 120s timeout），页数多时更明显。
- **模型能力**：弱模型可能在润色阶段改坏 JSON；若频繁解析失败，可考虑去掉润色或仅对 `title-bullets` 页做局部扩写。
- **后续可选**：将三阶段拆成可配置 `content_pipeline: single | multi`；或增加 `detailLevel` 请求参数控制 bullet 长度。

## 验证

```bash
curl -X POST http://localhost:3000/ppt/generate \
  -H "Content-Type: application/json" \
  -d '{"topic":"2025 年人工智能发展趋势","slideCount":8}' \
  -o ai-trends.pptx
```

打开 pptx，检查：要点是否含具体信息、是否出现 chart / two-column / 配图版式、末页是否为 full-image。
